## 1. TL;DR

Replace the empty-state `/top-searches` experience with corpus-derived default questions, generated by a **scheduled daily background worker** that clusters Qdrant content into topics and asks an LLM to produce natural questions per topic in all 8 locales. Results are persisted and blended with real crowdsourced trending at read time via weighted random selection. Resolve the caller's locale on GET endpoints via a `?locale=` → session memory → `Accept-Language` → `en` chain, and add a language switcher to geto-web.

**Generation is never triggered by a user request and never by ingest.** User queries only read pre-materialized rows.

**Why:** Cold-start buckets (new locales, fresh corpus, niche topics) return empty suggestion trays today. First-run UX is the highest-leverage moment in onboarding; an empty tray is a measurable drop-off.

**Impact:** New code in `kenjaku-service`, `kenjaku-infra`, `kenjaku-api`, `kenjaku-ingest`; one migration; one new Redis namespace; small frontend additions. No breaking API changes (`?locale=` stays as override).

## 2. Background

### 2.1 Objective

Give every user a meaningful, on-topic suggestion tray on first touch, in their own language, whether or not the corpus has crowdsourced signal yet.

### 2.2 Goals

1. `/top-searches` and `/autocomplete` always return ≥ K results per locale when the corpus has any documents, even with zero crowdsourced data.
2. Suggestions reflect the actual Qdrant corpus — ingesting new documents expands the default pool.
3. All 8 supported locales (`en`, `zh`, `zh-TW`, `ja`, `ko`, `de`, `fr`, `es`) have ≥ 10 defaults per bucket before the first real user lands.
4. Crowdsourced trending still influences ranking once signal exists — blend, don't replace.
5. Cascade delete: removing a document removes the defaults it spawned.
6. Locale preference on GET endpoints is resolved transparently; callers don't need to guess.
7. Frontend users can explicitly switch language.

### 2.3 Non-Goals

- Personalized-per-user defaults (no signal yet).
- Real-time / on-request generation (latency + cost).
- User-voted or admin-curated defaults.
- Legal/compliance curation layer (explicitly out per Sam).
- Tenant-scoped defaults for whitelabel (explicitly out per Sam).
- Cross-locale machine translation — we generate natively per locale, not MT from English.

## 4. Domain Design

### 4.1 Use Cases

| # | Actor | Scenario |
|---|---|---|
| UC-1 | Background worker | Once a day, fingerprints the Qdrant collection. If changed, clusters chunk embeddings into ~20 topics, asks the LLM to produce 3–5 questions per (topic, locale), writes to a staging batch, and atomically swaps into `default_suggestions`. |
| UC-2 | Background worker | If the fingerprint matches the previous run, skip generation entirely — no LLM calls, no writes. |
| UC-3 | End user | Calls `GET /top-searches` — server resolves locale, loads crowdsourced + pre-materialized defaults, returns a weighted-random blended pool. **No LLM call on the hot path.** |
| UC-4 | End user | Types a prefix into autocomplete — server prefix-matches crowdsourced and default rows, returns a blended list. |
| UC-5 | End user | Runs `POST /search` with a Japanese question; translator detects `ja`; server writes `sl:{session_id} = ja` in Redis so subsequent GETs inherit the locale. |
| UC-6 | End user | Toggles the language switcher on geto-web — client sends explicit `?locale=ja` on every GET. |
| UC-7 | Operator | Runs `kenjaku-ingest seed-refresh-now` to force an out-of-schedule regeneration (useful after large ingest batches or prompt tweaks). |

### 4.3 Blending Algorithm (read path)

Pseudo-code inside `SuggestionService::get_top(locale, limit)`:

```
crowd    = trending_repo.get_top(locale, Daily, LARGE_LIMIT, min_count)
defaults = default_repo.list_active_by_locale(locale, LARGE_LIMIT)  // status='Active' only

pool: Vec<(item, weight)> = []
for c in crowd:    pool.push( (c, c.search_count as f64) )
for d in defaults: pool.push( (d, d.weight as f64) )

if pool.is_empty(): return []

# Weighted random sampling without replacement (Efraimidis–Spirakis).
# Each item gets key = -ln(U) / weight where U ∈ (0,1); sort ascending; take first `limit`.
return sample_without_replacement(pool, limit)
```

Properties:
- Higher search_count → higher probability of surfacing, but never deterministic.
- Defaults always have some chance of appearing even when crowdsourced is hot.
- Caller sees variety across refreshes (UX win for sticky suggestion trays).
- `O(n)` with `n` = pool size, bounded by `LARGE_LIMIT = 50`.

For **autocomplete** (`search_popular`), the same blend runs over prefix-filtered rows, then weighted-samples down to `limit`.

### 4.4 Refresh Worker Algorithm (daily background job)

Scheduled via tokio in `kenjaku-server`, runs at 03:00 UTC. Lock via PG advisory lock (`pg_try_advisory_lock`) so multiple server replicas don't duplicate work.

```
fn daily_refresh():
    lock = pg.try_advisory_lock(SEED_REFRESH_LOCK_ID)
    if !lock: return  // another replica is running it

    # 1. Fingerprint the corpus
    stats = qdrant.collection_info(collection_name)
    fingerprint = sha256(collection_name || stats.points_count || stats.max_updated_at_usec)
    last = pg.select_latest_active_batch()
    if last.corpus_fingerprint == fingerprint:
        log.info("corpus unchanged, skipping refresh")
        return

    # 2. Start a new batch row (status=Running)
    batch_id = pg.insert refresh_batches { fingerprint, started_at=now, status=Running }

    # 3. Sample + cluster
    #    Pull up to SAMPLE_CAP (e.g. 2000) representative points from Qdrant with
    #    their vectors and payload title/summary. If the collection is smaller,
    #    pull all of it.
    samples = qdrant.scroll(collection_name, with_vector=true, with_payload=true, limit=SAMPLE_CAP)

    #    Cluster using mini-batch k-means over the embedding vectors.
    #    K is config-driven, default 20. Fewer than K*3 samples → reduce K.
    clusters = kmeans_minibatch(samples.vectors, k=min(K, samples.len / 3))

    #    Per cluster: pick the N=5 chunks closest to the centroid as representatives.
    #    These go into the LLM prompt as topical evidence.

    # 4. Per-cluster generation — ONE multi-locale call per cluster
    for cluster in clusters:
        excerpt = concat_top_n_chunks(cluster, n=5, max_chars=2000)
        result = llm.generate_cluster_questions(excerpt)  // returns { label, questions: {en: [...], zh: [...], ...} }
        for (locale, qs) in result.questions:
            kept = safety_filter(qs)                      // regex + length + dedup
            pg.insert default_suggestions {
                batch_id, locale, topic_cluster_id=cluster.id,
                topic_label=result.label, question, weight=DEFAULT_WEIGHT, ...
            }

    # Total LLM calls: K calls per run. K=20 → 20 calls on a run that actually happens.
    # Steady state (unchanged corpus): 0 calls — fingerprint short-circuits the worker.
    # Monthly estimate: 3-5 ingest days × 20 calls = ~100 calls/month. <$0.01 on Gemini Flash.

    # 5. Atomic swap
    pg.transaction:
        update refresh_batches set status='Active'     where id=batch_id
        update refresh_batches set status='Superseded' where id != batch_id and status='Active'
    # Read path queries WHERE status='Active', so the swap is instant and zero-downtime.

    # 6. Retention: drop batches older than RETENTION_BATCHES (default 3) after swap.
    pg.delete from default_suggestions where batch_id in (
        select id from refresh_batches
        order by started_at desc
        offset RETENTION_BATCHES
    )
```

### 4.5 Clustering Rationale

- **Why k-means over embeddings instead of per-document?** Per-document generation scales with corpus size (expensive) and produces narrow questions. Per-cluster scales with topic count (bounded, cheap) and produces cross-document thematic questions that actually match how users think ("how does staking work?" spans multiple docs).
- **Why mini-batch k-means?** `linfa-clustering` crate has a pure-Rust impl. For 2000 × 1536-dim vectors it runs in <1s. No Python/sklearn dependency.
- **Why sample cap?** Generation quality plateaus quickly; 2000 samples is enough to find the top 20 topics. Full-corpus clustering is unnecessary.
- **Fingerprint skip:** if nothing changed in Qdrant since the last successful run, we skip the entire worker — zero LLM cost on steady-state days.

### 4.6 Generation Prompt (one call per cluster, all 8 locales at once)

```
You generate search suggestion questions for a document search engine, for a
cluster of related document excerpts.

Excerpts:
{excerpt}

Task:
1. Produce a 3–5 word English topic label for this cluster.
2. For EACH of the 8 locales below, produce 3–5 natural questions a curious
   reader would plausibly ask and that the documents in this cluster could
   answer.

Locales: en (English), zh (简体中文), zh-TW (繁體中文), ja (日本語),
         ko (한국어), de (Deutsch), fr (Français), es (Español)

Rules (apply to ALL locales):
- Native, natural phrasing — not translated-sounding. A ja reader should feel
  a native ja speaker wrote the questions.
- Preserve product names, tickers, and proper nouns verbatim across locales.
- No financial advice, no price predictions, no "should I buy / sell" framing.
- No yes/no questions — prefer "what / how / why / when".
- Keep each question under 80 characters.

Output strict JSON:
{
  "label": "...",
  "questions": {
    "en":    ["...", "...", "..."],
    "zh":    ["...", "...", "..."],
    "zh-TW": ["...", "...", "..."],
    "ja":    ["...", "...", "..."],
    "ko":    ["...", "...", "..."],
    "de":    ["...", "...", "..."],
    "fr":    ["...", "...", "..."],
    "es":    ["...", "...", "..."]
  }
}
```

Enforced via Gemini `responseMimeType: application/json` + `responseSchema` with all 8 locale keys required.

**Why one call per cluster instead of one per (cluster, locale)?** Gemini handles multi-locale output well at the Flash tier — quality is on par with per-locale calls in practice, latency is lower, and it's 8× cheaper. If locale quality suffers in QA, we can fall back to the per-locale fan-out without any schema changes.

Post-filter:
- Regex deny-list: `/price|should i (buy|sell)|will .* hit|prediction|forecast/i` → drop.
- Length guard: 5–120 chars.
- Dedup (case-insensitive) within the same `(batch_id, locale, topic_cluster_id)`.

### 4.7 Locale Resolution Chain

Implemented as an Axum extractor `ResolvedLocale` in `crates/kenjaku-api/src/extractors/locale.rs`:

```
1. If `?locale=<bcp47>` present and parseable via DetectedLocale::from_bcp47 → use it.
2. Else if request carries `X-Session-Id` (or `?session_id=`) and Redis `sl:{id}` exists → use it.
3. Else if `Accept-Language` header present → parse via `accept-language` crate,
   map first supported tag to `Locale`.
4. Else → `Locale::En`.
```

Failures at any step fall through, not error.

### 4.8 Session Locale Memory

New service `crates/kenjaku-service/src/locale_memory.rs`:

```rust
pub struct LocaleMemory { redis: RedisClient }

impl LocaleMemory {
    pub async fn record(&self, session_id: &str, locale: Locale);     // fire-and-forget
    pub async fn lookup(&self, session_id: &str) -> Option<Locale>;
}
```

- Redis key: `sl:{session_id}` → bcp47 tag string
- TTL: 2 hours, refreshed on every `record` (sliding window)
- Written from `SearchService::search` and `search_stream` after translator detection
- Read from the `ResolvedLocale` extractor

## 5. Data Design

### 5.1 Migration: `migrations/20260408000001_default_suggestions.{up,down}.sql`

```sql
-- up
CREATE TABLE refresh_batches (
    id                  BIGSERIAL PRIMARY KEY,
    corpus_fingerprint  TEXT        NOT NULL,
    status              TEXT        NOT NULL CHECK (status IN ('running','active','superseded','failed')),
    started_at          TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    completed_at        TIMESTAMPTZ,
    llm_calls           INTEGER     NOT NULL DEFAULT 0,
    questions_kept      INTEGER     NOT NULL DEFAULT 0,
    questions_rejected  INTEGER     NOT NULL DEFAULT 0
);
CREATE INDEX idx_refresh_batches_status ON refresh_batches (status);

CREATE TABLE default_suggestions (
    id                BIGSERIAL PRIMARY KEY,
    batch_id          BIGINT      NOT NULL REFERENCES refresh_batches(id) ON DELETE CASCADE,
    locale            TEXT        NOT NULL,
    question          TEXT        NOT NULL,
    question_lower    TEXT        NOT NULL,  -- case-insensitive dedup + prefix
    topic_cluster_id  INTEGER     NOT NULL,
    topic_label       TEXT        NOT NULL,
    weight            INTEGER     NOT NULL DEFAULT 10,
    generated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE (batch_id, locale, topic_cluster_id, question_lower)
);

CREATE INDEX idx_default_suggestions_batch_locale ON default_suggestions (batch_id, locale);
CREATE INDEX idx_default_suggestions_prefix       ON default_suggestions (locale, question_lower text_pattern_ops);

-- down
DROP TABLE default_suggestions;
DROP TABLE refresh_batches;
```

Rationale:
- **Atomic swap via `refresh_batches.status`** — the read path filters on `status='active'`, the worker writes the new batch in `running` state, then in one transaction flips the previous `active` → `superseded` and the new `running` → `active`. Readers see either the old batch or the new one, never a half-written mix.
- `question_lower` enables ILIKE prefix matching without sequential scan.
- UNIQUE constraint is per-batch — rerunning a batch is impossible (new batch = new id), but intra-batch duplicates (LLM returning the same question twice) are blocked.
- Cascade delete on `batch_id` means the retention step (drop old batches) is a single `DELETE FROM refresh_batches`.

### 5.2 Privacy / Retention

- No PII stored — questions are LLM-generated against public corpus content.
- Rows live as long as their source document; cascade delete enforced in ingest delete path.
- Redis `sl:{session_id}` keys carry only a bcp47 tag; TTL 2 hours; no query text.

### 5.3 Capacity

- ~5 questions/doc × 8 locales = 40 rows/doc. 10K docs = 400K rows. Negligible.
- Per-locale in-memory pool cap `LARGE_LIMIT = 50` → <5 KB hot per request.
- Redis memory for session-locale: ~50 bytes × active sessions. 100K DAU ≈ 5 MB.

### 6.1 `GET /api/v1/top-searches`

**Before:**
```

```

**After:**
```
?locale=en          (optional override, highest priority)
?session_id=...     (optional; also honored via X-Session-Id header)

```

**Response (unchanged shape):**
```json
{
  "items": [
    { "query": "How does Level Up work?", "source": "default",       "weight": 10 },

  ],
  "resolved_locale": "ja",
  "resolved_locale_source": "session_memory"

```

Two new fields (`resolved_locale`, `resolved_locale_source`) for debuggability; additive, non-breaking.

### 6.2 `GET /api/v1/autocomplete`

Same locale resolution chain applied. Prefix-matches against both `popular_queries` and `default_suggestions`, blends via weighted random sample, returns `limit` items.

### 6.3 `POST /api/v1/search` (existing)

Side-effect added: after `resolve_translation`, call `locale_memory.record(req.session_id, detected_locale)` fire-and-forget. No request/response shape change.

### 6.4 Admin CLI (kenjaku-ingest)

New command:
```
kenjaku-ingest seed-refresh-now [--force] [--dry-run]
```

Connects to Postgres + Qdrant + LLM and triggers the same refresh procedure the daily worker runs. `--force` bypasses the fingerprint skip. Useful for:
- Initial population on a new deployment
- After a bulk ingest when operators don't want to wait for 03:00 UTC
- After tweaking the generation prompt

**Ingest pipeline is NOT modified.** `ingest url` / `ingest folder` only upsert to Qdrant — they never call the seed generator.

## 8. Security Design

| Concern | Mitigation |
|---|---|
| Prompt injection via ingested docs (doc tries to get LLM to emit `[evil.com](...)`) | Output passes the same regex safety filter; questions are <80 chars and plain text — no markdown links in the schema. |
| Financial-advice leakage | Prompt constraint + post-regex deny-list (`/price|should i buy/i` etc.). Monitored via WARN logs. |
| Session id guessing → locale poisoning | Worst case: attacker flips another user's suggestion tray language. No data leakage, TTL bounded. Acceptable. |
| `Accept-Language` header parsing panics on malformed input | Use `accept-language` crate (returns `Vec`, not panics). Wrap in graceful fallback to `en`. |
| SQL injection via prefix on autocomplete | Parameterized queries throughout (existing pattern). `question_lower` is generated server-side. |
| ReDoS on safety regex | Use `regex` crate (no backtracking). Patterns are anchored + bounded. |

No new auth surface. No new external calls on the hot path (reads are PG + Redis only; LLM lives in ingest path).

## 9. Compatibility Design

- `?locale=` query param remains fully functional on GETs — strictly additive resolution chain.
- `POST /search` continues to ignore `locale` (already removed).
- `GET /top-searches` response adds fields; clients ignoring unknown fields (geto-web does) keep working.

### 10.1 Rollout Plan

**Phase 1 — Infrastructure + first run (ship together)**
1. Merge migration `20260408000001_default_suggestions` (adds `refresh_batches` + `default_suggestions`).

**Phase 3 — Observability hardening**
9. Add Prometheus counters: `kenjaku_suggestions_served_total{locale, source}`.

### 10.3 Fallback / Kill Switches

- `default_suggestions.enabled: false` → refresh worker does not start; read path returns crowdsourced only (current behavior). Existing rows stay untouched.
- To revert to an earlier batch after a bad refresh: `UPDATE refresh_batches SET status='active' WHERE id=<prev>; UPDATE refresh_batches SET status='superseded' WHERE id=<bad>`. Retention keeps the last 3 batches for exactly this reason.
- `locale_memory.enabled: false` → extractor skips the Redis lookup step.
- Both flags hot-reloadable? No — restart required; config is loaded at startup today.

### 10.4 Monitoring

| Metric | Alert threshold |
|---|---|
| `kenjaku_seed_refresh_runs_total{status}` | `failed` > 0 in a day → PagerDuty (SEV3) |
| `kenjaku_seed_refresh_duration_seconds` | p95 > 600s (LLM hung or clustering blew up) |
| `kenjaku_seed_refresh_last_success_timestamp` | now - value > 48h → alert (job missed two runs) |
| `kenjaku_seed_refresh_llm_calls_total` | >> expected (K×9) → prompt bug or retry loop |
| `kenjaku_suggestions_served_total{source="default"}` per locale | stays at 0 for > 1h on non-empty corpus → misconfig |
| `kenjaku_locale_resolved_total{source}` | track ratio; `query_param` dominance = session memory broken |
| Redis `sl:*` key count | sanity only; bounded by session churn |

### 10.5 Testing

- **Unit:** blend algorithm (deterministic with seeded RNG), safety regex, locale extractor chain, fingerprint hasher, clusterer on fixture vectors, atomic-swap SQL.
- **Integration:**
  - Seeded Qdrant fixture → run `seed-refresh-now --force` → assert `refresh_batches.status='active'` + 8 × K × ~4 rows.
  - Re-run with same fingerprint → assert skip (no new batch, no LLM calls via mock provider).
  - Corrupt a batch mid-run (kill worker) → assert `status='failed'`, previous `active` still served.
  - GET `/top-searches` with empty `popular_queries` → assert all-defaults response.
  - GET with seeded crowdsourced → assert mixed pool.

### 10.7 Risks

| Risk | Severity | Mitigation |
|---|---|---|
| Gemini rate limits under backfill (8 locales × N docs) | Med | Throttle backfill CLI; sequential per-doc, parallel-across-locales with bounded concurrency (e.g. `futures::stream::iter().buffer_unordered(4)`) |
| Native ja/ko quality lower than en | Med | Spot-check before Phase 1 signoff; prompt explicitly asks for native phrasing |
| Blend RNG makes tests flaky | Low | Seeded RNG (`rand::rngs::StdRng::from_seed`) in tests |
| Session memory misattribution if session_id reused across users | Low | 7-day TTL + documented as "per device" |
| Crowdsourced row count out of sync with Redis (previous bug) | Low | Already fixed by `EXCLUDED.search_count` overwrite |
| Safety filter lets through something sketchy | Med | WARN log on every rejection; monitor sample manually first week |
| Accept-Language header spoofed | Low | No trust boundary crossed — it's just a UX hint |
