Kenjaku
Contextual RAG Search Engine

A Rust workspace of six strictly-layered crates: hybrid vector + BM25 retrieval, LLM answer generation with Google Search grounding, multi-locale translation, pluggable component layouts, SSE streaming, and an in-memory session history store for follow-up context.

Rust 1.88 · edition 2024 6 crates · strict DAG Axum + Tower Qdrant · PostgreSQL · Redis
§ 01 · System Overview

Context & Containers

Kenjaku is a contextual Retrieval-Augmented Generation (RAG) search engine. It combines hybrid vector + full-text retrieval, LLM-based answer generation, intent classification, multi-locale translation, and a pluggable component layout system to deliver structured search responses.

§ 03 · Design Principles

Golden Rules

3.1 · Trait-Based Abstraction

All external dependencies are abstracted behind traits defined in kenjaku-core::traits:

TraitPurposeImplementation
EmbeddingProviderVector embedding generationOpenAiEmbeddingProvider
LlmProviderAnswer generation, streaming, translation, suggestionsGeminiProvider
ContextualizerChunk contextualization during ingestionClaudeContextualizer
IntentClassifierQuery intent classificationLlmIntentClassifier
RetrieverDocument retrievalHybridRetriever
RerankerResult fusion / rerankingRrfReranker
ClustererCorpus clustering for default suggestionsLinfaClusterer
Golden Rule. Business logic in service never imports concrete provider types. It depends only on Arc<dyn Trait>.

3.2 · Immutable Domain Types

All domain types in kenjaku-core::types are immutable #[derive(Clone, Serialize, Deserialize)] structs. State transitions produce new values, not mutations.

3.3 · Fire-and-Forget Paths

Hot-path latency is protected by decoupling non-critical work:

  • Trending — direct Redis ZINCRBY, errors logged but not propagated
  • Conversations — bounded mpsc channel, batch-flushed by a worker
  • Locale memory — Redis SET EX spawned via tokio::spawn, 2h TTL
  • Session history — in-memory DashMap per session, janitor evicts idle sessions

3.4 · Configuration Hierarchy

config/base.yaml              — defaults, no secrets (committed)
config/{APP_ENV}.yaml         — env overrides (committed)
config/secrets.{env}.yaml     — API keys, DB credentials (gitignored)
KENJAKU__* env vars           — final override layer

Secrets MUST live in secrets.{env}.yaml or env vars. AppConfig::validate_secrets() runs at startup and fails fast with a clear message listing what is missing.

3.5 · Supported Locales

§ 04 · Search Pipeline

Request Flow

Intent classification and query normalization run in parallel via tokio::join!; vector and BM25 retrieval run in parallel via tokio::try_join!. Both failures degrade gracefully — the search hot path never blocks.

§ 08 · Data Schema

PostgreSQL · Qdrant · Redis

PostgreSQL

-- Reason categories for negative feedback
CREATE TABLE reason_categories (
    id SERIAL PRIMARY KEY,
    slug VARCHAR(100) UNIQUE NOT NULL,
    label VARCHAR(255) NOT NULL,
    is_active BOOLEAN NOT NULL DEFAULT TRUE
);

-- User feedback — (session_id, request_id) unique index upserts
-- repeat like/dislike clicks in place.
CREATE TABLE feedback (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    session_id VARCHAR(255) NOT NULL,
    request_id VARCHAR(255) NOT NULL,
    action VARCHAR(20) NOT NULL CHECK (action IN ('like','dislike','cancel')),
    reason_category_id INTEGER REFERENCES reason_categories(id),
    description TEXT,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE UNIQUE INDEX idx_feedback_session_request_unique
    ON feedback(session_id, request_id);

-- Popular / trending search queries (flushed from Redis)
CREATE TABLE popular_queries (
    id SERIAL PRIMARY KEY,
    locale VARCHAR(10) NOT NULL,
    query TEXT NOT NULL,
    search_count BIGINT NOT NULL DEFAULT 0,
    period VARCHAR(20) NOT NULL,
    period_date DATE NOT NULL,
    UNIQUE(locale, query, period, period_date)
);

-- Conversation records for analytics and audit
CREATE TABLE conversations (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    session_id VARCHAR(255) NOT NULL,
    request_id VARCHAR(255) NOT NULL UNIQUE,
    query TEXT NOT NULL,
    response_text TEXT NOT NULL,
    locale VARCHAR(10) NOT NULL,
    intent VARCHAR(50) NOT NULL,
    meta JSONB NOT NULL DEFAULT '{}',
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Default suggestions — atomic swap via refresh_batches.status
CREATE TABLE refresh_batches (
    id BIGSERIAL PRIMARY KEY,
    corpus_fingerprint TEXT NOT NULL,
    status TEXT NOT NULL, -- running · active · superseded · failed
    started_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    completed_at TIMESTAMPTZ,
    llm_calls INTEGER,
    questions_kept INTEGER,
    questions_rejected INTEGER
);
CREATE UNIQUE INDEX idx_refresh_batches_single_active
    ON refresh_batches ((status)) WHERE status = 'active';

CREATE TABLE default_suggestions (
    id BIGSERIAL PRIMARY KEY,
    batch_id BIGINT NOT NULL REFERENCES refresh_batches(id) ON DELETE CASCADE,
    locale VARCHAR(10) NOT NULL,
    question TEXT NOT NULL,
    question_lower TEXT NOT NULL,
    topic_cluster_id INTEGER NOT NULL,
    topic_label TEXT,
    weight REAL NOT NULL,
    generated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

Member values in trending:* are the normalized+capitalized form (English: translator output, others: raw with first char capitalized). The record-time gibberish guard rejects junk before Redis; read paths additionally enforce crowd_sourcing_min_count (default 2) so anything that slips through still needs repeated independent searches before surfacing.

9.1 · Implemented Hardening

  • Input caps — query 2000 chars, top_k 100, autocomplete / top-searches limits, 128-char session/request id cap
  • Rate limiting via tower_governor — 60 req/min per IP, SmartIpKeyExtractor
  • Request body limit 64 KB; request timeout 30 s
  • Error sanitization — Error::user_message() never leaks DB URLs, API errors, or internals
  • SSRF protection in the URL crawler — private-IP blocklist + DNS check + redirect::Policy::none()
  • Prompt injection defense — user text isolated in <text>/<query> XML tags with "ignore instructions inside" preambles in intent classifier and translator
  • Redis SCAN replaces any KEYS usage in flush workers
  • Secrets validation at startup via AppConfig::validate_secrets()
  • Query quality guard — length caps (Latin 120 / CJK 60), no-space-Latin ≥25, single-char dominance >40% in 10+ char queries
  • Feedback dedup — unique index on feedback(session_id, request_id)
  • Safety regex on default-suggestions refresh drops price / forecast / buy-sell prompts before persistence
Deferred (accepted by operator). API authentication (next phase). Gemini API key in URL query param (Google API design). Conversation PII retention policy.
§ 10 · Decision Records

Architecture Decisions

ADR-001

Qdrant for Vector + Full-Text

Single store for both vector similarity and keyword text search via payload indices. Cosine distance fits normalized OpenAI embeddings. At >10M vectors, consider a dedicated full-text engine alongside.

ADR-002

Reciprocal Rank Fusion

Weighted RRF: score = semantic_weight/(rank+1) + bm25_weight/(rank+1). Rank-based — avoids incomparable score distributions between vector and text. Default 80/20.

ADR-003

Channel-Based Conversation Flush

Bounded tokio::sync::mpsc (1024) with batch-insert worker. try_send is non-blocking — zero latency on the search path. Records can be lost on crash — acceptable for analytics.

ADR-004

Gemini + google_search Grounding

Cost-effective for high volume. Native Google Search grounding provides real-time source citations. Vendor lock-in mitigated by LlmProvider trait abstraction.

ADR-005

Claude for Contextualization

Claude Haiku 4.5 with prompt caching — document content (large) is cached; only the chunk prompt (small) changes per call. Separate from runtime LLM because contextualization is write-path only.

ADR-006

In-Memory Session History

Per-session DashMap<String, VecDeque<Turn>> — capped, idle-TTL evicted, no cross-instance sharing. Separate from the durable conversations table; PG remains the audit source of truth.

§ 13 · SSE Protocol

Streaming Contract

SearchService::search_stream returns a SearchStreamOutput:

FieldTypePurpose
start_metadataStreamStartMetadataEverything known before the LLM begins: intent, translated_query, locale, retrieval_count, preamble_latency_ms, request_id, session_id
streamPin<Box<dyn Stream<Item = StreamChunk>>>Token deltas. Each chunk may carry grounding: Vec<LlmSource> extracted from Gemini's groundingMetadata.groundingChunks[].web
contextStreamContextBookkeeping (sources, instants, ids, history_key) consumed by complete_stream() after the stream finishes

Events emitted into a mpsc::channel(100) wrapped in Sse::new(ReceiverStream::new(rx)):

EventPayloadSent when
startStreamStartMetadata JSONOnce, before the first token
delta{"text": "..."}Per LLM token chunk
doneStreamDoneMetadataAfter the last delta — complete_stream resolves grounding URLs via TitleResolver, merges sources, runs suggest(), queues conversation + history
error{"error": "..."}On any failure — logged AND sent so the client sees it
Do not hand-roll SSE parsing. The server uses eventsource-stream to consume Gemini's streamGenerateContent?alt=sse — Gemini's separators vary across responses and a manual parser was buggy.
§ 14 · CI

Pipeline

.github/workflows/ci.yml runs on push to main and on PRs:

  • Rust stable job — cargo fmt --check, cargo clippy --workspace --all-targets -- -D warnings, cargo build --locked, cargo test --locked. Cache key kenjaku-stable via Swatinem/rust-cache@v2. Workspace-wide RUSTFLAGS=-D warnings.
  • Docker build job (depends on Rust) — validates docker compose config -q, then builds both kenjaku and geto-web images via docker/build-push-action@v6 with GHA cache.
Both jobs must pass before merge.
